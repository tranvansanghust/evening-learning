# SQLAlchemy ORM Models Documentation

## Overview
Complete SQLAlchemy 2.0 ORM models for the intelligent learning system database layer. All models are in `backend/app/models/` and inherit from `Base` defined in `app.database`.

## Models Created

### 1. User (`user.py`)
Represents user accounts with Telegram integration.

**Key Columns:**
- `user_id` (PK, auto-increment)
- `telegram_id` (unique, indexed)
- `username` (nullable)
- `level` (0-3 for beginner to expert)
- `created_at`, `updated_at` (UTC timestamps)

**Relationships:**
- `user_courses` → list of enrolled courses
- `quiz_sessions` → list of quiz sessions
- `quiz_answers` → list of answers submitted

### 2. Course (`course.py`)
Represents available courses (from Udemy or internal source).

**Key Columns:**
- `course_id` (PK, auto-increment)
- `name`, `description` (Text)
- `source` ('udemy', 'internal', etc.)
- `source_id` (external ID from source platform)
- `total_lessons` (count)
- `created_at` (UTC timestamp)

**Relationships:**
- `lessons` → all lessons in course
- `user_courses` → all enrollments

**Indexes:**
- Composite: `(source, source_id)` for quick external lookups

### 3. UserCourse (`user_course.py`)
Many-to-many relationship with metadata between users and courses.

**Key Columns:**
- `user_course_id` (PK)
- `user_id` (FK → users)
- `course_id` (FK → courses)
- `status` (PASS, FAIL, IN_PROGRESS)
- `started_at`, `completed_at` (UTC timestamps, completed_at nullable)

**Relationships:**
- `user` → parent User
- `course` → parent Course
- `quiz_summaries` → all summaries for this enrollment

**Cascade:** Delete-on-cascade for both FKs

### 4. Lesson (`lesson.py`)
Individual lesson within a course.

**Key Columns:**
- `lesson_id` (PK)
- `course_id` (FK → courses)
- `sequence_number` (order in course)
- `title`, `description`
- `content_url` (for frontend rendering)
- `estimated_duration_minutes` (nullable)

**Relationships:**
- `course` → parent Course
- `concepts` → learning objectives in lesson
- `quiz_sessions` → quizzes for this lesson

**Indexes:**
- `course_id` (FK)
- Composite: `(course_id, sequence_number)` for ordered retrieval

### 5. Concept (`concept.py`)
Key learning objective within a lesson.

**Key Columns:**
- `concept_id` (PK)
- `lesson_id` (FK → lessons)
- `name` (concept title)
- `description` (detailed explanation)

**Relationships:**
- `lesson` → parent Lesson
- `quiz_answers` → answers related to this concept

### 6. QuizSession (`quiz_session.py`)
A single quiz/oral-test session for a user.

**Key Columns:**
- `session_id` (PK)
- `user_id` (FK → users)
- `lesson_id` (FK → lessons)
- `status` ('active', 'completed')
- `messages` (JSON) → conversation history array:
  ```json
  [
    {"role": "bot", "content": "Question...?"},
    {"role": "user", "content": "User answer..."},
    ...
  ]
  ```
- `started_at`, `completed_at` (UTC, completed_at nullable)

**Relationships:**
- `user` → User taking quiz
- `lesson` → Lesson being tested
- `quiz_answers` → individual Q&A pairs
- `quiz_summary` → one-to-one post-quiz summary

**Indexes:**
- `user_id`, `lesson_id` (FKs)
- `status` (for active session queries)
- `started_at` (for timeline queries)

### 7. QuizAnswer (`quiz_answer.py`)
Individual question-answer pair within a quiz.

**Key Columns:**
- `answer_id` (PK)
- `session_id` (FK → quiz_sessions)
- `concept_id` (FK → concepts)
- `question` (Text) - the question asked
- `user_answer` (Text) - student's response
- `is_correct` (Boolean) - LLM evaluation
- `engagement_level` ('low', 'medium', 'high')
- `created_at` (UTC timestamp)

**Relationships:**
- `session` → parent QuizSession
- `concept` → Concept being tested

**Indexes:**
- `session_id`, `concept_id` (FKs)
- `is_correct` (for mastery filtering)

### 8. QuizSummary (`quiz_summary.py`)
Post-quiz summary generated by LLM.

**Key Columns:**
- `summary_id` (PK)
- `session_id` (FK → quiz_sessions, unique) - one summary per session
- `user_course_id` (FK → user_courses, nullable)
- `concepts_mastered` (JSON) - array of mastered concept names
- `concepts_weak` (JSON) - array of objects:
  ```json
  [
    {
      "concept": "concept_name",
      "user_answer": "what user said",
      "correct_explanation": "correct answer"
    },
    ...
  ]
  ```
- `created_at` (UTC timestamp)

**Relationships:**
- `quiz_session` → parent QuizSession
- `user_course` → optional UserCourse for progress tracking

**Indexes:**
- `session_id` (unique)
- `user_course_id` (FK)
- `created_at` (for timeline queries)

## Usage Examples

### Import Models
```python
from app.models import (
    User, Course, UserCourse, Lesson, Concept,
    QuizSession, QuizAnswer, QuizSummary
)
from app.database import SessionLocal
```

### Create New User
```python
db = SessionLocal()
new_user = User(
    telegram_id="123456789",
    username="john_doe",
    level=0
)
db.add(new_user)
db.commit()
```

### Enroll User in Course
```python
enrollment = UserCourse(
    user_id=user.user_id,
    course_id=course.course_id,
    status="IN_PROGRESS"
)
db.add(enrollment)
db.commit()
```

### Create Quiz Session with Conversation
```python
session = QuizSession(
    user_id=user.user_id,
    lesson_id=lesson.lesson_id,
    status="active",
    messages=[
        {"role": "bot", "content": "What is a variable?"},
        {"role": "user", "content": "It's a container for data"}
    ]
)
db.add(session)
db.commit()
```

### Add Quiz Answer
```python
answer = QuizAnswer(
    session_id=session.session_id,
    concept_id=concept.concept_id,
    question="What is a variable?",
    user_answer="It's a container for data",
    is_correct=True,
    engagement_level="high"
)
db.add(answer)
db.commit()
```

### Generate Quiz Summary
```python
summary = QuizSummary(
    session_id=session.session_id,
    user_course_id=enrollment.user_course_id,
    concepts_mastered=["variables", "data_types"],
    concepts_weak=[
        {
            "concept": "functions",
            "user_answer": "Not sure",
            "correct_explanation": "Functions are reusable blocks of code..."
        }
    ]
)
db.add(summary)
db.commit()
```

### Query User's Courses
```python
user_enrollments = db.query(UserCourse)\
    .filter(UserCourse.user_id == user.user_id)\
    .all()

for enrollment in user_enrollments:
    print(f"Course: {enrollment.course.name}")
    print(f"Status: {enrollment.status}")
```

### Query Quiz Sessions with Answers
```python
session = db.query(QuizSession)\
    .filter(QuizSession.session_id == session_id)\
    .first()

print(f"Session Status: {session.status}")
print(f"Quiz Answers: {len(session.quiz_answers)}")
print(f"Summary: {session.quiz_summary}")
```

### Get User's Learning Progress
```python
user = db.query(User)\
    .filter(User.telegram_id == "123456789")\
    .options(joinedload(User.user_courses))\
    .first()

for uc in user.user_courses:
    print(f"Course: {uc.course.name}")
    print(f"Progress: {uc.status}")
    if uc.completed_at:
        print(f"Completed: {uc.completed_at}")
```

## Data Flow

### Learning Flow
1. User enrolls in Course → `UserCourse` (status: IN_PROGRESS)
2. User studies Lesson → lesson content served from `content_url`
3. User starts quiz → `QuizSession` created (status: active)
4. User answers question → `QuizAnswer` stored
5. Quiz completes → `QuizSession` updated (status: completed)
6. LLM generates summary → `QuizSummary` created
7. Progress updated → `UserCourse` status changes if quiz passed

### Progress Tracking
- User level (0-3) managed in `User.level`
- Course progress via `UserCourse.status` and `completed_at`
- Concept mastery tracked via `QuizSummary.concepts_mastered`
- Weak areas for review in `QuizSummary.concepts_weak`

## Performance Considerations

### Indexes
- All foreign keys indexed for JOIN operations
- Status columns indexed for filtering (PASS/FAIL/IN_PROGRESS, active/completed)
- Timestamp columns indexed for range queries
- Composite indexes for common multi-column queries

### JSON Columns
- `quiz_sessions.messages` stores full conversation history
- `quiz_summaries.concepts_mastered/weak` for LLM summary results
- MySQL supports native JSON queries and indexing

### Relationships
- Using `lazy="joined"` for frequently accessed relationships
- Using `lazy="select"` for large collections to avoid N+1 queries
- Cascade delete configured to maintain referential integrity

## Migration to Database

# Apply migration
alembic upgrade head
```

```

## Notes

- All timestamps are UTC (server-side defaults)
- Cascade deletes prevent orphaned records
- Two-way relationships via `back_populates` enable navigation in both directions
- JSON columns support flexible schema for LLM outputs
- Models follow SQLAlchemy 2.0 best practices
- Each model file is self-contained and fully documented
